\section{Results}
\label{sec:results}

\subsection{Transmit Path}

We can see several interesting trends in the TX case. Clearly looking at figure \ref{fig:tx_tp} turning the vIOMMU on without vFree is catastrophic, the throughput degrades by orders of magnitude. However with vFree on the TX path achieves parity with the same setup without vIOMMU protections and it does this without any dependence on the number of VMs. So vFree works perfectly, attaining optimal throughput while providing protections in multi-tenant environments. One thing is worth discussing in more detail, the case in which the vIOMMU is on but without the vFree strategy shows increasing bandwidth as a function of the number of VMs. This means that even though one VM is not able to saturate the bandwidth of the machine multiple VMs are still able to share the bandwidth quite well.

\subsection{Receive Path}

The RX path is more disappointing than the TX path. From figure \ref{fig:rx_tp} although vFree achieves optimal throughput for a single VM the throughput decreases as a function of the number of VMs for all methods. By doubling the number of VMs the throughput is decreased by roughly half. What is interesting here is that all the methods decrease and roughly even out in terms of throughput for more VMs although vIOMMU off does show a slight edge. With the throughput of each method being decreased roughly equally we have to conclude that the fault of the degradation in throughput has little to no dependence on the page free strategy used.

\subsection{Receive Path Optimization}
\label{subsec:rx_opt}

After seeing the results of the experiment, we wanted to investigate the cause of the RX bottleneck. First, we wanted to remove some implementation differences between the RX and TX benchmarks. These changes were relatively minor, such as making the warmup time before experiments are preformed the same and removing some pre-experiment setup operations done in RX that are not preformed in TX. These changes provided minor improvements to the RX throughput, but did not change the overall trends. Next, we enabled aRFS~\ref{subsubsec:arfs}, which again improved throughput but did not change overall trends.

To help explore the problem, we collected flamegraphs and measured NIC drop rates during the RX experiments. First, we put artificial limits on the clients to prevent them from overwhelming the system. This minimally helped throughput but we still saw excessive drop rates. The drop rates were minimized by enabling flow control~\ref{subsubsec:flowctrl}, but this did not improve throughput. Finally, we attempted to alter the buffer sizes used in the RX experiments, such as the receive buffer size, netdev budget, and MSI buffer size. These changes also did not improve throughput.
